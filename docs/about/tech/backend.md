# Backend

## Node.js / TypeScript

![NestJS](../../images/nest.svg){ class="tech-logo" }
### NestJS
[https://nestjs.com/](https://nestjs.com/){ class="tech-link" }

- **Quando usar:** quer arquitetura mais guiada.
- **Pontos fortes:** DI, módulos, guards, pipes, Swagger pronto, testes bem suportados.
- **Pontos de atenção:** curva inicial (decorators, módulos).

![Express](../../images/express-js.jpg){ class="tech-logo" }
### Express (TS)
[https://expressjs.com/](https://expressjs.com/){ class="tech-link" }

- **Quando usar:** quer simplicidade e controle total.
- **Pontos fortes:** leve, rápido de começar.
- **Pontos de atenção:** precisa estruturar do zero (padrões, camadas, validação).

## Python

![FastAPI](../../images/fastapi.svg){ class="tech-logo" }
### FastAPI 
[https://fastapi.tiangolo.com/](https://fastapi.tiangolo.com/){ class="tech-link" }

- **Quando usar:** quer produtividade + tipagem + OpenAPI automático.
- **Pontos fortes:** docs automáticas (Swagger), performance boa, validação com Pydantic.
- **Pontos de atenção:** organização de projeto e camadas devem ser definidas.

## Ruby

![Rails](../../images/rails.png){ class="tech-logo-h" }
### Rails (API mode) 
[https://rubyonrails.org/](https://rubyonrails.org/){ class="tech-link" }

- **Quando usar:** quer produtividade máxima e convenções fortes.
- **Pontos fortes:** scaffolding rápido, validação, migrations, bom com PostgreSQL/MySQL.
- **Pontos de atenção:** menos comum no ecossistema atual de microsserviços (mas super válido).
